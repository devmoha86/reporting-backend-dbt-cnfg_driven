Reporting Dashboard Framework — Code & Execution Reference

Every file explained, every step documented. Smoke-tested end to end against a synthetic DuckDB mart and a live local Redis before this was written.

FastAPI DuckDB / Athena Redis cache-aside Config-driven dbt

🎯 What This Solves

Each of the 18 reporting dashboards needs the same shape of backend: a filtered/paginated queue table, a clickable status-summary chart, dropdown filter lookups, and a CSV export — all cached in Redis, all reading from DuckDB locally / Athena in production. Hand-writing that per dashboard means 18× the FastAPI endpoints, cache-key logic, and dbt boilerplate.

This framework turns each dashboard into a YAML config. One generic router factory reads the config and produces the live endpoints; one generator script reads the config and scaffolds the dbt models. Adding dashboard #2 means writing a config file, not new Python.

📍 Where New Dashboard SQL Logic Goes

The one rule that matters most in this whole document: dashboard-specific business logic — SLA thresholds, cycle-time math, custom rollups, and chart aggregations — is written in dbt SQL, never in the shared Python framework code. There are exactly two places it goes:

1. backend/dashboards/<dashboard_id>/dbt/marts/mart_<dashboard_id>_queue.sql, inside the TODO lines that generate_dbt_models.py stamps in for every column you flagged needs_clarification: true in your config.yaml.
2. backend/dashboards/<dashboard_id>/dbt/marts/mart_<dashboard_id>_summary.sql, which is the entire definition of that dashboard's summary chart. router_factory.py never aggregates anything — it only does SELECT * FROM this model. Whatever shape your chart needs (status counts, sums, a trend line) is written here, in SQL.
FileWho edits itWhat goes there
dashboards/<id>/config.yamlYou, by handSource table, staging columns, primary key, status values, filters, API shape — declarative, no SQL.
dashboards/<id>/dbt/staging/stg_<id>.sqlGenerator only — never hand-editPure column casting. If a column is wrong, fix config.yaml and re-run the generator; don't patch this file directly, the next re-run silently overwrites it.
dashboards/<id>/dbt/marts/mart_<id>_queue.sqlYou, inside the TODO blocks onlyThis is it. Every line that starts , NULL AS <col> -- TODO (NEEDS CLARIFICATION): ... is a placeholder for real SQL — replace the NULL with the actual expression. The plain column list above the TODO block is generator output; leave it alone.
dashboards/<id>/dbt/marts/mart_<id>_summary.sqlYou, freely — this is the other place business logic goesThe generator writes a zero-filled status-count default here, but this file is the entire business logic for that dashboard's summary chart. The API layer (router_factory.get_summary()) does nothing but SELECT * FROM this model and return the rows — no GROUP BY, no chart-shaping, no aggregation in Python, ever. If your dashboard's summary chart is a sum-by-category, a trend line, or a multi-series rollup instead of a status count, rewrite this file's query to produce that shape directly. Re-running the generator overwrites it back to the default, same caveat as the queue mart's TODO blocks.
dashboards/<id>/dbt/marts/schema_<id>.ymlGenerator onlydbt tests (not_null/unique on the primary key, accepted_values on the status field) plus a documented entry for every other column.
Why the TODO block survives a re-run but nothing else does: build_mart_queue_sql() in the generator always emits NULL AS <col> for everything still listed under mart.derived_columns in the config — it does not read the file back first. If you've already filled in real SQL there and then re-run the generator (e.g. because you added a new staging column), your business logic will be silently overwritten back to NULL. Either avoid re-running the generator once the TODOs are filled in, or re-paste the logic back in afterward. See Step 4 below.

🔄 Request Data Flow

GET /api/ldc/case-queue?servicer_number=7721
  ↓ router_factory's get_queue() (built from dashboards/ldc/config.yaml)
  ↓ make_cache_key("ldc", "case-queue", {...}) → cache_get()
  ↓ Redis HIT  → return cached JSON directly
  ↓ Redis MISS → build_filter_clause() → execute_query() → DuckDB/Athena
  ↓ cache_set() writes the result back before returning it
  ↓ JSON response → Angular MFE

The summary-chart flow is deliberately identical in shape, with one difference worth calling out: there's no build_filter_clause() step and no GROUP BY anywhere in Python. get_summary() reads the dbt-built mart_<id>_summary table and nothing else.

GET /api/ldc/status-summary
  ↓ router_factory's get_summary() (config.mart.summary_model_name)
  ↓ make_cache_key("ldc", "status-summary") → cache_get()
  ↓ Redis HIT  → return cached JSON directly
  ↓ Redis MISS → execute_query("SELECT * FROM mart_ldc_status_summary") — that's it, no aggregation here
  ↓ cache_set() writes {"title", "items", "count"} back before returning it
  ↓ JSON response → Angular MFE (chart-shaping, if any, happens client-side from raw rows)

🪜 The Full Execution Pipeline — Step by Step, With Code

This is the sequence every dashboard goes through, start to finish, from an empty folder to a live, cached API endpoint. Each step names the exact file involved and what runs inside it. This is the section to follow when you've been handed a new dashboard to build independently of the other 17.

Step 1 — Copy the template folder
Every dashboard starts as a copy of dashboards/_template/. Because the framework discovers dashboards by folder (see Step 6), copying the folder is the entire setup — there's no registry file to edit.
cp -r backend/dashboards/_template backend/dashboards/<dashboard_id>
Step 2 — Fill in dashboards/<dashboard_id>/config.yaml
Declare the source table, the staging columns (with casts), the mart's primary key / status field / status values, any derived_columns whose business logic isn't known yet, and the API shape (filters, sort, dropdowns, export). No SQL, no Python — see the full annotated template under Dashboard Configs below.
Validated automatically: config_loader.load_dashboard_config() parses this file straight into config_schema.DashboardConfig (a Pydantic model). A typo'd field name or invalid op/refresh_tier value fails immediately with a clear validation error — not a silent bug discovered later at request time.
Step 3 — Run the generator
scripts/generate_dbt_models.py reads config.yaml and writes three SQL files plus one schema file, all inside that same dashboard's folder — nothing is written anywhere else in the repo.
cd backend
python scripts/generate_dbt_models.py dashboards/<dashboard_id>/config.yaml
# From generate_dbt_models.py's main() — output path is derived from the config's own location:
dashboard_dir = cfg_path.parent                      # dashboards/<dashboard_id>/
staging_dir = dashboard_dir / "dbt" / "staging"
marts_dir = dashboard_dir / "dbt" / "marts"

(marts_dir / f"{cfg['mart']['queue_model_name']}.sql").write_text(build_mart_queue_sql(cfg, config_filename))
(marts_dir / f"{cfg['mart']['summary_model_name']}.sql").write_text(build_mart_summary_sql(cfg, config_filename))
(marts_dir / f"schema_{cfg['dashboard_id']}.yml").write_text(build_schema_yml(cfg, config_filename))

Produces: dashboards/<id>/dbt/staging/stg_<id>.sql, dashboards/<id>/dbt/marts/mart_<id>_queue.sql, .../mart_<id>_summary.sql, .../schema_<id>.yml. Full generated output for ldc is under Generated Output.

Step 4 — Fill in the TODO business logic ⭐ the step that's actually dashboard-specific
Open dashboards/<dashboard_id>/dbt/marts/mart_<dashboard_id>_queue.sql. Everything above the TODO block is generated column passthrough — leave it. Each TODO line is a derived_columns entry from your config; replace the NULL with the real expression.
-- dashboards/ldc/dbt/marts/mart_ldc_case_queue.sql, as generated:
SELECT
    request_no,
    request_status,
    ...
    attribute_count
    , NULL AS queue_age  -- TODO (NEEDS CLARIFICATION): Elapsed-time calc TBD - calendar vs business hours
    , NULL AS cycle_time  -- TODO (NEEDS CLARIFICATION): Business-day calc TBD; only meaningful once Completion Date is set
    , NULL AS sla_status  -- TODO (NEEDS CLARIFICATION): SLA thresholds (Within SLA / At Risk / Breached) TBD
FROM {{ ref('stg_ldc_case_requests') }}
ORDER BY request_no
-- Same file, after a developer fills in the business rule (illustrative example):
    , DATE_DIFF('day', submission_date, COALESCE(completion_date, CURRENT_DATE)) AS queue_age
    , CASE WHEN completion_date IS NOT NULL
           THEN DATE_DIFF('day', submission_date, completion_date) END AS cycle_time
    , CASE WHEN sla_due_date IS NULL THEN NULL
           WHEN completion_date IS NOT NULL AND completion_date <= sla_due_date THEN 'Within SLA'
           WHEN completion_date IS NULL AND CURRENT_DATE > sla_due_date THEN 'Breached'
           WHEN completion_date IS NULL THEN 'At Risk'
           ELSE 'Breached' END AS sla_status
Don't re-run the generator after this step expecting it to keep your edits — it doesn't read the file back, it overwrites it. If the config changes later (a new staging column, say), re-running will reset every TODO line back to NULL; re-apply the business logic by hand afterward. See Where New Dashboard SQL Logic Goes.
Step 5 — dbt run / dbt test
Builds the staging + mart tables in the warehouse and runs the generated tests (not_null/unique on the primary key, accepted_values on the status field).
cd backend/dbt_project   # once dbt_project.yml/profiles.yml exist — see "Running dbt" below
dbt run  --select <dashboard_id>
dbt test --select <dashboard_id>
This package doesn't bundle dbt_project.yml/profiles.yml/sources.yml (see Running dbt). Step 7 — Anatomy of a live request
What actually runs when the MFE calls your new endpoint — the full chain through router_factory.py, every piece of it generic and config-driven.
GET {base_path}/{queue_endpoint}?{your filter param}=...
  1. router_factory's get_queue() collects query params declared in api.filters
  2. make_cache_key(dashboard_id, queue_endpoint, params)        → core/cache/keys.py
  3. cache_get(key)                                              → core/cache/client.py
       HIT  → return cached JSON, done
       MISS ↓
  4. build_filter_clause(api.filters, params) → positional WHERE + params  → core/filters.py
  5. execute_query(sql, params)                                  → core/db/connection.py (duckdb/athena)
  6. cache_set(key, result, ttl=cache_ttl_seconds)                → core/cache/client.py
  7. JSON response

Every numbered line is generic framework code your dashboard never touches — see core/router_factory.py for the full implementation.

🧩 Core Framework

backend/core/config_schema.py
The contract every dashboard YAML must satisfy, enforced by Pydantic. If a config is missing a required field or uses an invalid op/refresh_tier value, it fails to load with a clear validation error instead of failing silently at request time.
"""
Defines the schema every dashboard config YAML must follow.
Adding a new dashboard = writing a new YAML file that validates against this schema,
not writing new Python code. See backend/dashboards/_template/config.yaml to start one.
"""
from __future__ import annotations
from typing import Optional, Literal
from pydantic import BaseModel, Field

RefreshTier = Literal["near_real_time", "snapshot_eod", "snapshot_eom", "needs_clarification"]
FilterOp = Literal["eq", "ilike", "gte", "lte"]


class SourceConfig(BaseModel):
    database: str          # Glue/Iceberg database in the source data team's account (placeholder until confirmed)
    table: str              # Iceberg table backing the case/request grain
    loan_attributes_table: Optional[str] = None  # Iceberg table backing loan-level rollups, if any


class StagingColumn(BaseModel):
    name: str               # column name after staging (snake_case)
    type: str                # DuckDB/Athena-compatible type, e.g. varchar, date, integer
    source_col: str          # raw column name in the source table


class StagingConfig(BaseModel):
    model_name: str
    columns: list[StagingColumn]


class DerivedColumn(BaseModel):
    """A mart column whose business logic isn't known yet (e.g. SLA thresholds)."""
    name: str
    needs_clarification: bool = False
    note: Optional[str] = None


class MartConfig(BaseModel):
    queue_model_name: str
    summary_model_name: str
    primary_key: str
    status_field: str
    status_values: list[str] = Field(default_factory=list)
    derived_columns: list[DerivedColumn] = Field(default_factory=list)


class FilterDef(BaseModel):
    field: str               # mart column name
    param: str                # query-param name exposed in the API
    op: FilterOp = "eq"


class SortDef(BaseModel):
    field: str
    direction: Literal["asc", "desc"] = "desc"


class ApiConfig(BaseModel):
    base_path: str                    # e.g. /api/ldc
    queue_endpoint: str = "queue"
    summary_endpoint: str = "summary"
    filters: list[FilterDef] = Field(default_factory=list)
    default_page_size: int = 25
    default_sort: SortDef
    filter_lookup_fields: list[str] = Field(default_factory=list)
    export_enabled: bool = True


class DashboardConfig(BaseModel):
    dashboard_id: str                 # short slug, used in routes/cache keys/dbt model names
    display_name: str
    description: str = ""
    refresh_tier: RefreshTier
    cache_ttl_seconds: int = 3600
    source: SourceConfig
    staging: StagingConfig
    mart: MartConfig
    api: ApiConfig

SourceConfigCross-account Glue/Iceberg database + table — still placeholders pending the source data team.

backend/core/config_loader.py
Discovers and validates every dashboard config by scanning folders, not files — each dashboard is dashboards/<dashboard_id>/config.yaml. Folders prefixed with _ (the template) are skipped on purpose so they never get registered as a live dashboard.
from pathlib import Path
import yaml
from .config_schema import DashboardConfig


def load_dashboard_config(path: Path) -> DashboardConfig:
    data = yaml.safe_load(Path(path).read_text())
    return DashboardConfig(**data)


def load_all_dashboards(dashboards_dir: Path) -> list[DashboardConfig]:
    """Loads dashboards/<dashboard_id>/config.yaml for every dashboard folder,
    except folders prefixed with '_' (templates/examples)."""
    configs = []
    for d in sorted(Path(dashboards_dir).iterdir()):
        if not d.is_dir() or d.name.startswith("_"):
            continue
        config_file = d / "config.yaml"
        if config_file.exists():
            configs.append(load_dashboard_config(config_file))
    return configs
backend/core/db/connection.py
The same swap point already established for this project. DATA_LAYER=duckdb for local/dev, DATA_LAYER=athena for production. execute_query()'s signature and return shape never change — every dashboard router calls only this function.
"""
THE swap point. Identical to the pattern already established for this project:
DATA_LAYER=duckdb  -> local DuckDB file (Phase 1 / local dev)
DATA_LAYER=athena  -> pyathena against real AWS Athena (Phase 2 / production)

execute_query()'s signature and return shape never change. Every dashboard router
calls only this function - no dashboard-specific code touches the database directly.
"""
import os

_DATA_LAYER = os.environ.get("DATA_LAYER", "duckdb")
_conn = None


def get_connection():
    global _conn
    if _conn is not None:
        return _conn

    if _DATA_LAYER == "duckdb":
        import duckdb
        path = os.environ.get(
            "DUCKDB_PATH",
            os.path.join(os.path.dirname(__file__), "..", "..", "fnma.duckdb"),
        )
        _conn = duckdb.connect(path, read_only=False)

    elif _DATA_LAYER == "athena":
        import pyathena
        _conn = pyathena.connect(
            region_name=os.environ["AWS_REGION"],
            s3_staging_dir=os.environ["ATHENA_S3_STAGING"],
            schema_name=os.environ.get("ATHENA_SCHEMA", "reporting"),
        )
    else:
        raise RuntimeError(f"Unknown DATA_LAYER: {_DATA_LAYER!r} (expected 'duckdb' or 'athena')")

    return _conn


def execute_query(sql: str, params: list | None = None) -> list[dict]:
    """Executes parameterised SQL ($1, $2, ... positional). Returns list of row dicts."""
    conn = get_connection()
    if _DATA_LAYER == "duckdb":
        rel = conn.execute(sql, params or [])
        cols = [d[0] for d in rel.description]
        return [dict(zip(cols, r)) for r in rel.fetchall()]
    else:
        cursor = conn.cursor()
        cursor.execute(sql, params or [])
        cols = [d[0] for d in cursor.description]
        return [dict(zip(cols, r)) for r in cursor.fetchall()]
backend/core/cache/client.py
Fail-open Redis client: if Redis is unreachable, cache_get returns None and cache_set silently no-ops, so endpoints fall through to the database transparently. BUG FIXED HERE — see "Bug Found & Fixed" below.
"""
Same fail-open Redis pattern already established for this project: if Redis is
unreachable, cache_get returns None and cache_set/invalidate silently no-op, so
endpoints fall through to the database transparently. No dashboard ever has to
handle a Redis outage itself.
"""
import os
import json
import datetime
from decimal import Decimal

REDIS_URL = os.environ.get("REDIS_URL", "redis://localhost:6379/0")
_pool = None


def _json_default(obj):
    """DuckDB/Athena return datetime.date and Decimal for DATE/NUMERIC columns,
    neither of which json.dumps handles by default. Without this, cache_set's
    fail-open except-clause swallows the TypeError and silently never caches
    any row containing a date - exactly the columns most dashboards have."""
    if isinstance(obj, (datetime.date, datetime.datetime)):
        return obj.isoformat()
    if isinstance(obj, Decimal):
        return float(obj)
    raise TypeError(f"Object of type {type(obj).__name__} is not JSON serializable")


async def get_redis():
    global _pool
    if _pool is None:
        import redis.asyncio as aioredis
        _pool = aioredis.from_url(
            REDIS_URL,
            decode_responses=True,
            socket_connect_timeout=2,
            socket_timeout=2,
        )
    return _pool


async def cache_get(key: str):
    try:
        r = await get_redis()
        raw = await r.get(key)
        return json.loads(raw) if raw else None
    except Exception:
        return None


async def cache_set(key: str, value, ttl: int = 3600):

Deterministic cache key builder — sorted params so identical filter sets always map to the same key, regardless of the order query params arrived in.
def make_cache_key(dashboard_id: str, endpoint: str, params: dict | None = None) -> str:
    """
    Deterministic cache key, same convention used elsewhere in this project:
    sorted params so identical filter sets always map to the same key.
    Example: make_cache_key("ldc", "case-queue", {"servicer_number": "SVC-001", "page": 1})
             -> "ldc:case-queue:page=1:servicer_number=SVC-001"
    """
    parts = [dashboard_id, endpoint]
    if params:
        for k in sorted(params):
            v = params[k]
            if v is None or v == "":
                continue
            parts.append(f"{k}={v}")
    return ":".join(parts)

Generic version of the project's existing _filter_clause() pattern — the field list comes from config instead of being hand-written per endpoint. Always positional $1, $2, ... — never f-string user input into SQL.
"""
Generic version of the _filter_clause() pattern already used in this project's
backend, except the field list comes from each dashboard's config instead of
being hand-written per endpoint. Always positional ($1, $2, ...) - never f-string
user input into SQL.
"""
from .config_schema import FilterDef

_OP_SQL = {
    "eq": "=",
    "ilike": "ILIKE",
    "gte": ">=",
    "lte": "<=",
}


def build_filter_clause(filter_defs: list[FilterDef], query_params: dict) -> tuple[str, list]:
    conditions: list[str] = []
    params: list = []
    for fd in filter_defs:
        value = query_params.get(fd.param)
        if value is None or value == "":
            continue
        params.append(f"%{value}%" if fd.op == "ilike" else value)
        conditions.append(f"{fd.field} {_OP_SQL[fd.op]} ${len(params)}")
    where = ("WHERE " + " AND ".join(conditions)) if conditions else ""
    return where, params
backend/core/router_factory.py
The piece that removes per-dashboard coding. Given a validated DashboardConfig, returns a fully wired FastAPI router — queue table, status summary, filter-lookup dropdowns, CSV export — all using the cache-aside pattern above. Adding dashboard #2 means writing a new YAML config, not a new router module.
The summary endpoint below is deliberately dumb: get_summary() does SELECT * FROM {summary_mart} and nothing else. It used to recompute a status-count GROUP BY against the queue mart in Python — duplicating logic dbt already computed, hardcoding a bar-chart shape, and never actually touching the dbt-built mart_<id>_summary table at all. That's fixed: all aggregation now lives in mart_<id>_summary.sql (see Where New Dashboard SQL Logic Goes), and this function just passes its rows through, the same way get_queue() passes through the queue mart's rows.
"""
This is the piece that removes the per-dashboard coding: given a validated
DashboardConfig, it returns a fully wired FastAPI router (queue table, status
summary, filter-lookup dropdowns, CSV export) using the cache-aside pattern
already established in this project. Adding dashboard #2 means writing a new
YAML config, not a new router module.

Summary endpoint is intentionally dumb: it does SELECT * FROM the dbt-built
summary mart and returns the rows as-is - no GROUP BY, no aggregation, no
chart-shaping logic lives here. Every dashboard's summary chart can be a
completely different aggregation (status counts, sums by category, a trend
line, ...) because that logic is written once in that dashboard's own
mart__summary.sql, not in this shared Python file. See
dashboards//dbt/marts/mart__summary.sql.
"""
import csv
import io

from fastapi import APIRouter, Query, Request
from fastapi.responses import StreamingResponse

from .config_schema import DashboardConfig
from .db.connection import execute_query
from .cache.client import cache_get, cache_set
from .cache.keys import make_cache_key
from .filters import build_filter_clause


def build_dashboard_router(config: DashboardConfig) -> APIRouter:
    router = APIRouter(prefix=config.api.base_path, tags=[config.dashboard_id])
    mart = config.mart.queue_model_name
    summary_mart = config.mart.summary_model_name
    sort_field = config.api.default_sort.field
    sort_dir = config.api.default_sort.direction.upper()

    def _collect_params(request: Request) -> dict:
        return {fd.param: request.query_params.get(fd.param) for fd in config.api.filters}

    # ---- Queue table -------------------------------------------------
    @router.get(f"/{config.api.queue_endpoint}", summary=f"{config.display_name} — queue")
    async def get_queue(
        request: Request,
        page: int = Query(1, ge=1),
        page_size: int = Query(config.api.default_page_size, ge=1, le=500),
    ):
        params_in = _collect_params(request)
        cache_key = make_cache_key(
            config.dashboard_id, config.api.queue_endpoint,
            {**params_in, "page": page, "page_size": page_size},
        )
        cached = await cache_get(cache_key)
        if cached is not None:
            return cached

        where, params = build_filter_clause(config.api.filters, params_in)
        offset = (page - 1) * page_size
        sql = (
            f"SELECT * FROM {mart} {where} "
            f"ORDER BY {sort_field} {sort_dir} "
            f"LIMIT ${len(params) + 1} OFFSET ${len(params) + 2}"
        )
        rows = execute_query(sql, params + [page_size, offset])

        count_sql = f"SELECT COUNT(*) AS n FROM {mart} {where}"
        total = execute_query(count_sql, params)[0]["n"]

        result = {"items": rows, "count": total, "page": page, "page_size": page_size}
        await cache_set(cache_key, result, ttl=config.cache_ttl_seconds)
        return result

    # ---- Summary chart (dumb passthrough) ------------------------------
    # No GROUP BY, no aggregation, no chart-shaping here. The mart__summary
    # model already contains whatever rollup that dashboard's chart needs -
    # this just selects it and hands the rows back untouched, exactly the way
    # get_queue() above hands back the queue mart's rows untouched. If a new
    # dashboard's chart is a sum-by-category, a trend line, or anything else,
    # that logic is written once in that dashboard's mart__summary.sql -
    # this function never has to change.
    @router.get(f"/{config.api.summary_endpoint}", summary=f"{config.display_name} — summary chart")
    async def get_summary():
        cache_key = make_cache_key(config.dashboard_id, config.api.summary_endpoint)
        cached = await cache_get(cache_key)
        if cached is not None:
            return cached

        rows = execute_query(f"SELECT * FROM {summary_mart}")
        result = {
            "title": f"{config.display_name} — Summary",
            "items": rows,
            "count": len(rows),
        }
        await cache_set(cache_key, result, ttl=config.cache_ttl_seconds)
        return result

    # ---- Filter / dropdown lookups -------------------------------------
    for field in config.api.filter_lookup_fields:
        def _make_lookup(field_name: str):
            async def lookup():
                cache_key = make_cache_key(config.dashboard_id, f"filters:{field_name}")
                cached = await cache_get(cache_key)
                if cached is not None:
                    return cached
                sql = (
                    f"SELECT DISTINCT {field_name} AS value FROM {mart} "
                    f"WHERE {field_name} IS NOT NULL ORDER BY {field_name}"
                )
                rows = execute_query(sql)
                result = {"items": [r["value"] for r in rows], "count": len(rows)}
                await cache_set(cache_key, result, ttl=max(config.cache_ttl_seconds, 21600))
                return result
            return lookup

        router.add_api_route(
            f"/filters/{field}", _make_lookup(field), methods=["GET"],
            summary=f"{config.display_name} — distinct values for {field}",
        )

    # ---- CSV export (same columns as the queue view) --------------------
    if config.api.export_enabled:
        @router.get(f"/{config.api.queue_endpoint}/export", summary=f"{config.display_name} — export CSV")
        async def export_csv(request: Request):
            params_in = _collect_params(request)
            where, params = build_filter_clause(config.api.filters, params_in)
            sql = f"SELECT * FROM {mart} {where} ORDER BY {sort_field} {sort_dir}"
            rows = execute_query(sql, params)

            buf = io.StringIO()
            if rows:
                writer = csv.DictWriter(buf, fieldnames=list(rows[0].keys()))
                writer.writeheader()
                writer.writerows(rows)
            buf.seek(0)
            return StreamingResponse(
                iter([buf.getvalue()]), media_type="text/csv",
                headers={"Content-Disposition": f"attachment; filename={config.dashboard_id}_export.csv"},
            )

    return router

backend/scripts/generate_dbt_models.py
Reads one dashboard config and scaffolds its dbt staging model, mart models, and schema.yml. Column casting, model structure, and test stubs are generated. Dashboard-specific business logic is left for a human in two places: the queue mart's TODO comments (anything flagged needs_clarification), and the summary mart, which is generated with a working zero-filled status-count default but is fully expected to be rewritten per dashboard for whatever chart aggregation it actually needs.

Usage

"""
Reads one dashboard YAML config and scaffolds its dbt staging model, mart models,
and schema.yml — written into that SAME dashboard's own folder
(dashboards/<dashboard_id>/dbt/staging, dashboards/<dashboard_id>/dbt/marts).
This is the "without much coding" part for the dbt layer: column casting, model
structure, and test stubs are generated; dashboard-specific business logic is
left for a human to fill in - as a TODO in the queue mart (flagged
needs_clarification in the config), and as a fully editable default in the
summary mart, since every dashboard's chart aggregation is different.

Usage:
    python scripts/generate_dbt_models.py dashboards/<dashboard_id>/config.yaml
"""
import sys
from pathlib import Path
import yaml


def _header(config_filename: str) -> str:
    return (
        f"-- AUTO-GENERATED by scripts/generate_dbt_models.py from {config_filename}\n"
        f"-- Re-run the generator after editing the config rather than hand-editing this file,\n"
        f"-- except inside the TODO blocks below, which need dashboard-specific business logic.\n\n"
    )


def build_staging_sql(cfg: dict, config_filename: str) -> str:
    cols = cfg["staging"]["columns"]
    lines = [f"    CAST({c['source_col']} AS {c['type'].upper()}) AS {c['name']}" for c in cols]
    body = ",\n".join(lines)
    src = cfg["source"]
    return (
        _header(config_filename)
        + f"SELECT\n{body}\nFROM {{{{ source('{src['database']}', '{src['table']}') }}}}\n"
    )


def build_mart_queue_sql(cfg: dict, config_filename: str) -> str:
    stg = cfg["staging"]["model_name"]
    col_names = [c["name"] for c in cfg["staging"]["columns"]]
    select_cols = ",\n".join(f"    {c}" for c in col_names)

    todo_lines = []
    for dc in cfg["mart"].get("derived_columns", []):
        tag = "TODO (NEEDS CLARIFICATION)" if dc.get("needs_clarification") else "TODO"
        note = dc.get("note", "")
        todo_lines.append(f"    , NULL AS {dc['name']}  -- {tag}: {note}")
    todo_block = ("\n".join(todo_lines) + "\n") if todo_lines else ""

    return (
        _header(config_filename)
        + f"SELECT\n{select_cols}\n{todo_block}FROM {{{{ ref('{stg}') }}}}\n"
        + f"ORDER BY {cfg['mart']['primary_key']}\n"
    )


def build_mart_summary_sql(cfg: dict, config_filename: str) -> str:
    """
    Generates a *default* summary mart: a zero-filled count of status_field,
    good enough for a simple bar/pie "how many requests in each status" chart.

    This default is a starting point, not the final word. Unlike staging/queue
    models, the summary mart is meant to be hand-edited per dashboard: a
    different dashboard's summary chart might be a sum-by-category, a trend
    over time, or a multi-series rollup, and that logic belongs only here, in
    SQL - never in the API layer, which just does SELECT * FROM this model.
    Re-running the generator overwrites hand edits, same caveat as the TODO
    blocks in mart_<id>_queue.sql.
    """
    stg = cfg["staging"]["model_name"]
    status_field = cfg["mart"]["status_field"]
    status_values = cfg["mart"].get("status_values", [])

    note = (
        "-- DEFAULT CHART LOGIC - EDIT FREELY.\n"
        "-- This is a generated starting point, not generator-protected output. The\n"
        "-- query below is this dashboard's entire summary-chart business logic: the\n"
        "-- API only does `SELECT * FROM` this model and returns the rows as-is, so\n"
        "-- whatever shape/aggregation this dashboard's chart needs (status counts,\n"
        "-- sums by category, a trend line, ...) is written here, once, in SQL - not\n"
        "-- in Python. Re-running the generator WILL overwrite any edits you make\n"
        "-- below; once this file diverges from the default, stop re-running the\n"
        "-- generator for this dashboard's summary mart.\n\n"
    )

    if status_values:
        values_rows = ",\n        ".join(f"('{v}')" for v in status_values)
        body = (
            f"with counts as (\n"
            f"    select\n        {status_field},\n        count(*) as request_count\n"
            f"    from {{{{ ref('{stg}') }}}}\n    group by {status_field}\n),\n"
            f"all_statuses as (\n"
            f"    select column1 as {status_field}\n"
            f"    from (values\n        {values_rows}\n    ) as t\n)\n"
            f"select\n"
            f"    s.{status_field},\n"
            f"    coalesce(c.request_count, 0) as request_count\n"
            f"from all_statuses s\n"
            f"left join counts c using ({status_field})\n"
            f"order by s.{status_field}\n"
        )
    else:
        body = (
            f"select\n    {status_field},\n    count(*) as request_count\n"
            f"from {{{{ ref('{stg}') }}}}\ngroup by {status_field}\n"
        )

    return _header(config_filename) + note + body


def build_schema_yml(cfg: dict, config_filename: str) -> str:
    pk = cfg["mart"]["primary_key"]
    status_field = cfg["mart"]["status_field"]
    status_values = cfg["mart"].get("status_values", [])
    queue_model = cfg["mart"]["queue_model_name"]
    summary_model = cfg["mart"]["summary_model_name"]
    values_yaml = "\n".join(f'                - "{v}"' for v in status_values)

    staging_col_names = [c["name"] for c in cfg["staging"]["columns"]]
    derived_cols = cfg["mart"].get("derived_columns", [])

    column_blocks = []
    for name in staging_col_names:
        if name == pk:
            column_blocks.append(f"      - name: {pk}\n        tests: [not_null, unique]")
        elif name == status_field:
            column_blocks.append(
                f"      - name: {status_field}\n"
                f"        tests:\n"
                f"          - accepted_values:\n"
                f"              values:\n"
                f"{values_yaml}"
            )
        else:
            column_blocks.append(f"      - name: {name}")
    for dc in derived_cols:
        note = dc.get("note", "")
        suffix = f"  # NEEDS CLARIFICATION: {note}" if dc.get("needs_clarification") else ""
        column_blocks.append(f"      - name: {dc['name']}{suffix}")
    columns_yaml = "\n".join(column_blocks)

    return f"""# AUTO-GENERATED by scripts/generate_dbt_models.py from {config_filename}
version: 2
models:
  - name: {queue_model}
    description: "{cfg['display_name']} — one row per {pk}."
    columns:
{columns_yaml}
  - name: {summary_model}
    description: "{cfg['display_name']} — summary chart data. Default is zero-filled status counts; this dashboard may have hand-edited the SQL for a different aggregation."
    columns:
      - name: {status_field}

"""

🔁 Cache Sync

backend/cache_sync/sync.py
Generalizes the project's existing cache-sync pattern: instead of a hardcoded mart list, this iterates every dashboard config and syncs it according to its declared refresh_tier. Run after dbt run — hourly via EventBridge for near_real_time dashboards, daily/monthly for snapshot dashboards.
Same fix as router_factory.py: sync_dashboard() used to recompute the status-count GROUP BY itself, against the queue mart, in Python. It now caches SELECT * FROM the summary mart directly — identical contract to get_summary(), so a Redis HIT after cache-sync and a Redis MISS hitting the API directly return the exact same shape.
"""
Generalizes the cache-sync pattern already established for this project: instead
of a hardcoded MARTS list, this iterates every dashboard config and syncs it
according to its declared refresh_tier. Run after `dbt run` in CodeBuild/CI -
near_real_time dashboards on an hourly EventBridge schedule, snapshot dashboards
on a daily/monthly one.

Dashboards with refresh_tier: needs_clarification are skipped on purpose - they
should not be silently scheduled until the cadence question is resolved.
"""
import asyncio
import sys
from pathlib import Path

sys.path.insert(0, str(Path(__file__).parent.parent))

from core.config_loader import load_all_dashboards
from core.config_schema import DashboardConfig
from core.db.connection import execute_query
from core.cache.client import cache_set, cache_invalidate_prefix
from core.cache.keys import make_cache_key

DASHBOARDS_DIR = Path(__file__).parent.parent / "dashboards"


async def sync_dashboard(cfg: DashboardConfig):
    mart = cfg.mart.queue_model_name
    summary_mart = cfg.mart.summary_model_name
    await cache_invalidate_prefix(cfg.dashboard_id)

    sort_field = cfg.api.default_sort.field
    sort_dir = cfg.api.default_sort.direction.upper()
    rows = execute_query(f"SELECT * FROM {mart} ORDER BY {sort_field} {sort_dir}")

    page_size = cfg.api.default_page_size
    queue_key = make_cache_key(cfg.dashboard_id, cfg.api.queue_endpoint, {"page": 1, "page_size": page_size})
    await cache_set(
        queue_key,
        {"items": rows[:page_size], "count": len(rows), "page": 1, "page_size": page_size},
        ttl=cfg.cache_ttl_seconds,
    )

    # Dumb passthrough, same contract as router_factory.get_summary(): just
    # cache whatever the dbt-built summary mart already computed. No
    # aggregation is recomputed here - that logic lives only in
    # mart__summary.sql, once, per dashboard.
    summary_rows = execute_query(f"SELECT * FROM {summary_mart}")
    summary_key = make_cache_key(cfg.dashboard_id, cfg.api.summary_endpoint)
    await cache_set(
        summary_key,
        {
            "title": f"{cfg.display_name} — Summary",
            "items": summary_rows,
            "count": len(summary_rows),
        },
        ttl=cfg.cache_ttl_seconds,
    )

    print(f"[cache-sync] {cfg.dashboard_id}: {len(rows)} rows cached (tier={cfg.refresh_tier})")


async def sync_all():
    for cfg in load_all_dashboards(DASHBOARDS_DIR):
        if cfg.refresh_tier in ("near_real_time", "snapshot_eod", "snapshot_eom"):
            await sync_dashboard(cfg)
        else:
            print(f"[cache-sync] SKIPPED {cfg.dashboard_id} — refresh_tier={cfg.refresh_tier}, needs clarification before scheduling")


if __name__ == "__main__":
    asyncio.run(sync_all())
